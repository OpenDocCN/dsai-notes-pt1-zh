## 中断操作的可能性

在深入代码之前，让我们先了解使用中断时可以进行的几种操作。

![](img/2ca1053d68b09d1570d2825fe5775056_5.png)

以下是几种核心的中断后操作：
1.  **恢复**：在不修改状态的情况下，根据用户输入继续执行。这正是我们将在本例中使用的操作。
2.  **更新并恢复**：更新图的状态，然后继续执行。
3.  **回退/时间旅行**：利用检查点机制，回到执行历史中的某个先前状态，并从那里重新开始或选择不同的分支。
4.  **分支**：从当前执行状态创建一个新分支，以探索不同的路径。
5.  **取消**：完全取消当前的执行。

![](img/2ca1053d68b09d1570d2825fe5775056_7.png)

![](img/2ca1053d68b09d1570d2825fe5775056_8.png)

这些操作提供了在图中断时控制流程的多种方式。

## 代码实现：带中断的图

现在，让我们通过一个简单的例子来看看如何具体实现中断。我们将使用之前提到的A->B->(C/D)结构。

以下是实现代码的关键部分：
```python
# 导入必要模块，包括中断方法
from langgraph.types import interrupt

![](img/2ca1053d68b09d1570d2825fe5775056_10.png)

![](img/2ca1053d68b09d1570d2825fe5775056_11.png)

# 定义节点B的函数
def node_b(state):
    # ... 执行节点B的其他逻辑 ...
    # 在此处设置中断点，等待人工输入
    human_decision = interrupt(“你想去节点C还是节点D？请输入C或D：”)
    # 根据输入更新状态并决定下一个节点
    state[‘path’] += ‘B’
    if human_decision == ‘C’:
        return {‘next’: ‘node_c’}
    else:
        return {‘next’: ‘node_d’}
```
我们创建了一个名为`resume.py`的文件来实现此图。代码结构与之前类似，但关键区别在于我们为图配置了**记忆存储**。

为什么需要记忆存储？当中断发生时，图会退出执行。记忆存储（或检查点）会记录中断发生的位置和当前状态，这样当我们稍后恢复时，就能从准确的中断点继续。

## 执行流程解析

上一节我们设置了图的结构和中断点，本节我们来模拟它的执行过程。

如果调用这个图，执行流程如下：
1.  打印“节点A”，状态更新。
2.  进入节点B，打印“节点B”。
3.  执行到`interrupt`方法，图暂停并退出。此时，系统会输出提示信息：“你想去节点C还是节点D？请输入C或D：”。
4.  中断点的所有信息（状态、下一个待执行节点等）都被保存在检查点中。

此时，执行流在节点B内部被“冻结”，等待外部指令来恢复。

## 恢复执行

当图在节点B中断后，我们需要一种方法来让它继续运行。这通过`Command`类来实现。

![](img/2ca1053d68b09d1570d2825fe5775056_16.png)

![](img/2ca1053d68b09d1570d2825fe5775056_17.png)

![](img/2ca1053d68b09d1570d2825fe5775056_18.png)

我们可以使用`app.invoke_command`方法并传入特定的关键字参数来恢复图。我们传递给该方法的数值（例如字母‘C’或‘D’）将被送入之前中断的`interrupt`方法调用处，作为人工决策的输入。

例如，如果我们传入`{“resume”: “D”}`，图将从节点B的中断点恢复，接收输入“D”，然后根据节点B的逻辑跳转到节点D，最终完成执行。

![](img/2ca1053d68b09d1570d2825fe5775056_20.png)

## 时间旅行的概念

![](img/2ca1053d68b09d1570d2825fe5775056_21.png)

![](img/2ca1053d68b09d1570d2825fe5775056_22.png)

![](img/2ca1053d68b09d1570d2825fe5775056_23.png)

![](img/2ca1053d68b09d1570d2825fe5775056_24.png)

在基本恢复中，一旦图执行完毕，线程就结束了。如果我们想用同一个线程ID回到之前的决策点（比如最初选了D，现在想改选C），就需要“时间旅行”。

这之所以可能，是因为LangGraph在每个节点执行后都会创建**检查点**。我们可以获取历史中的某个检查点，并从那里重新开始执行，就好像时光倒流一样。这为调试或探索不同决策路径提供了强大支持。我们将在下一节详细探讨这个概念。
