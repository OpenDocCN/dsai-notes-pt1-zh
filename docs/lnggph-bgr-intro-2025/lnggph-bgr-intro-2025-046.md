## 核心状态与逻辑

现在，让我们向上滚动，看看我们在 `page.tsx` 中做了什么。我们在这里维护了一个消息列表。例如，“How can I help you”这条消息是我们硬编码在这里的。我们为它分配了一个 `id`，并设置了 `isUser` 属性为 `false`，因为这是 AI 发出的消息。`type` 属性被设置为 `"message"`。未来可能会有不同类型的数据填充到这里，例如搜索查询或搜索结果，相应地，`type` 也会不同，以便我们显示不同的 UI。

![](img/12a8250f0073ca250115187b4188ade1_30.png)

![](img/12a8250f0073ca250115187b4188ade1_31.png)

![](img/12a8250f0073ca250115187b4188ade1_32.png)

在深入之前，让我展示一下这个消息对象可能包含的其他属性。这是一个 TypeScript 接口定义：

```typescript
interface Message {
  id: number;
  content: string;
  isUser: boolean;
  type: string;
  isLoading: boolean;
  searchInfo?: {
    stage: string;
    query?: string;
    urls?: string[];
  };
}
```

![](img/12a8250f0073ca250115187b4188ade1_34.png)

![](img/12a8250f0073ca250115187b4188ade1_35.png)

![](img/12a8250f0073ca250115187b4188ade1_36.png)

*   `id` 和 `content` 我们已经见过。
*   `isUser` 标识消息发送者。
*   `type` 标识消息类型。
*   `isLoading`：在请求发送到服务器后、大语言模型开始生成内容之前，我们会显示一个“正在输入”的 UI。这个状态就由 `isLoading` 控制。
*   `searchInfo`：它包含 `stage`（阶段）、`query`（查询）和 `urls`（网址）。阶段可以是“正在搜索网络”、“正在阅读”或“正在编写”等。我们维护这些信息是为了以特定的方式展示 UI。

![](img/12a8250f0073ca250115187b4188ade1_37.png)

例如，当我询问“下一次 SpaceX 发射是什么时候”时，你会看到“正在搜索网络”作为一个阶段出现，然后是“正在阅读”，最后是“正在编写”。所有这些都将作为一条 AI 消息响应的一部分。我们可以这样理解：它们都属于同一条消息。

## 核心事件处理：handleSubmit

![](img/12a8250f0073ca250115187b4188ade1_39.png)

![](img/12a8250f0073ca250115187b4188ade1_40.png)

`handleSubmit` 方法是 99% 逻辑所在的地方。当我们输入内容并按下回车时，就会触发这个事件。

以下是 `handleSubmit` 方法的主要步骤：

1.  **获取用户输入**：`currentMessage` 存储用户当前输入的内容。
2.  **创建新消息ID**：获取现有消息列表中的最大 `id`，然后加 1。
3.  **更新用户消息状态**：立即将用户的新消息添加到消息列表中。这是一个同步操作，所以消息会立刻显示在 UI 上。
4.  **清空输入框**：清空输入框，以便用户输入下一条消息。
5.  **处理 AI 响应流**：在 `try` 代码块中，处理从服务器接收的流式数据。
    *   首先，创建一个 AI 响应消息对象，设置 `isLoading: true` 以显示加载状态。
    *   构造请求 URL，指向我们后端的 API 端点（`localhost:8000`），并附上用户输入。
    *   使用浏览器的 **`EventSource` API** 连接服务器发送事件（Server-Sent Events）的端点。这是客户端接收流式数据的关键。
    *   为 `EventSource` 设置事件监听器。每当从服务器端接收到事件时，回调函数就会被触发。

## 处理服务器发送的事件

在 `EventSource` 的事件处理器中，我们根据接收到的数据 `type` 进行不同的处理：

*   **`checkpoint` 类型**：将检查点 ID 存储在客户端（例如使用 `useEffect`），以便在用户提出后续问题时，可以将其附加到请求参数中，这样服务器就知道是哪个会话，并能从中恢复记忆。
*   **`content` 类型**：将接收到的数据块追加到 `streamedContent` 变量中，并更新状态，以便实时反映在 UI 上。
*   **`search_start` 类型**：获取搜索查询，并更新消息状态中的 `searchInfo`。
*   **`search_results` 类型**：获取搜索结果的 URL 列表，并更新消息状态。
*   **`end` 类型**：接收到结束事件后，立即关闭 `EventSource` 连接。
