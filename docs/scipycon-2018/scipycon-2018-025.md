# 25：Astropy 的实际应用 🚀

在本节课中，我们将学习天文学 Python 库 Astropy 在两个实际生产环境中的应用：Zwicky 暂现源巡天（ZTF）和起源空间望远镜（OST）的模拟工具。我们将了解 Astropy 如何帮助解决天体测量、坐标转换、单位管理和宇宙学计算等关键问题。

## 什么是 Astropy？🔭

![](img/6f69786f046b1c5ffed49a1153d767d1_3.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_5.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_7.png)

Astropy 项目是一个社区驱动的项目，旨在汇集对天文学家有用的功能。它由一个包含基础功能的核心包和一个扩展的附属包生态系统组成。附属包需要满足特定标准才能获得认证。

![](img/6f69786f046b1c5ffed49a1153d767d1_9.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_11.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_13.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_15.png)

以下是本次课程中会涉及的一些 Astropy 子包：
*   **`io.fits`**：用于处理天文学中常用的灵活图像传输标准（FITS）文件。
*   **`table`**：用于处理表格数据。
*   **`coordinates`**：用于处理天球坐标。
*   **`wcs`**：代表世界坐标系，用于将像素映射到天空中的天球坐标。
*   **`units` 和 `constants`**：用于物理单位与常量的管理。
*   **`time`**：用于处理时间。
*   **`cosmology`**：用于宇宙学计算。

## Zwicky 暂现源巡天（ZTF）简介

![](img/6f69786f046b1c5ffed49a1153d767d1_27.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_28.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_30.png)

上一节我们介绍了 Astropy 的基本构成，本节中我们来看看它在 ZTF 项目中的具体应用。ZTF 是一个在过去三个月内刚刚启动的巡天项目。

![](img/6f69786f046b1c5ffed49a1153d767d1_32.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_34.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_35.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_37.png)

ZTF 是一个在加利福尼亚南部帕洛玛天文台进行的机器人巡天项目。在漫长的冬夜，每晚最多可获取 750 次曝光。每次曝光覆盖 47 平方度。相机由 16 个 CCD 组成，每个 CCD 为 6000 x 6000 像素，并有四个读出通道。因此，我们拥有一个 6.5 亿像素的相机，覆盖了巨大的视场。

![](img/6f69786f046b1c5ffed49a1153d767d1_39.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_40.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_42.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_44.png)

我们每晚处理数万张图像。原始图像数据每晚约 1 TB。我们每晚已经产生数十万到上百万个警报。这一切都在加州理工学院校园内的 IPAC 数据中心以全自动方式运行。我们在山上快门关闭后 10 分钟内处理完 95% 的图像。

![](img/6f69786f046b1c5ffed49a1153d767d1_46.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_48.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_50.png)

![](img/6f69786f046b1c5ffed49a1153d767d1_52.png)

ZTF 是 Zwicky 暂现源设施，这是我们的新相机。PTF（帕洛玛暂现源设施）是 ZTF 的前身，它使用了来自加拿大-法国-夏威夷望远镜的不同相机。即将到来的是大型综合巡天望远镜（LSST），目前正在南美洲建造，将在三四年后开始巡天。

ZTF 的视场比 LSST 大得多。但 LSST 拥有更大的望远镜（8 米口径）和 32 亿像素的相机。因此，ZTF 的规模大约是 LSST 的 10%。但我们使用的是现有的望远镜设施，所以这个项目只花费了约 2200 万美元，约为 LSST 的 10%。

ZTF 的主要科学目标是搜索暂现源。以下三张图像展示了在一个警报数据包中看到的内容：新的科学曝光图像（New）、参考曝光图像（REF，由之前多晚的曝光平均而成）以及减影图像（SUB），其中显示了可能是超新星等天体。

ZTF 的科学案例包括超新星、引力波事件、超大质量黑洞附近的潮汐瓦解事件、变星、小行星和彗星等。我们实际上正在通过华盛顿大学处理的警报机制来演练 LSST 的警报系统。

## 使用 Astropy 解决 ZTF 的天体测量问题

上一节我们了解了 ZTF 项目的概况，本节中我们来看看 Astropy 如何帮助我们解决其中的一个难题：为数据处理管线获取精确的天体测量结果。

我们使用 SExtractor 及其相关软件套件。SExtractor 是天文学中非常流行的源提取工具。SCAMP 是与之配套的天体测量求解器。SWARP 可以在获得正确的畸变解后对图像进行重映射。

我们在 ZTF 的前身 PTF 中使用了 SCAMP，但遇到了问题。例如，在图像的大部分区域天体测量解很好，但在角落会出现双星或三重星。这对于图像减影管线会造成严重干扰，产生大量虚假目标。

![](img/6f69786f046b1c5ffed49a1153d767d1_54.png)

我们的解决方案是构建一个天体测量先验信息提供给 SCAMP 求解器，以帮助其找到好的解。我们选取少量曝光进行一次性离线拟合。我们匹配具有像素坐标的恒星，并使用来自盖亚任务（欧洲空间局的天体测量任务）的精确赤经和赤纬。

![](img/6f69786f046b1c5ffed49a1153d767d1_56.png)

我们使用 Astropy 的世界坐标系（WCS），它通常将像素映射到天球坐标。但我们创建了一个理想化的、像素为 1 度的 WCS，从而能够将赤经和赤纬转换为切平面投影坐标 C 和 Aida。经过一些拟合后，我们可以得到像素偏移量（CRPIX1 和 CRPIX2）、CD 矩阵以及焦平面上 64 个象限图像各自的中心度偏移量。

![](img/6f69786f046b1c5ffed49a1153d767d1_58.png)

我们在切投影平面上进行计算。我们将盖亚星表转换回 C 和 Aida 切投影平面（单位为度）。另一方面，我们从 SExtractor 获取检测到的 X、Y 像素坐标，并进行转换，然后拟合一个线性矩阵和一些像素偏移的方程。理想情况下它们应该匹配，但现实中存在畸变，这些畸变在切投影平面上是二维多项式。SCAMP 求解器就是处理这些，并将结果表示在图像的元数据中。

我们使用 Astropy 方便地管理 SCAMP 的输入头文件。我们将这些先验信息与望远镜指向结合，生成一个包含畸变先验的头文件给 SCAMP。我们还输入 SExtractor 检测星表和盖亚天体测量星表给 SCAMP。SCAMP 会生成一个包含畸变解的头文件，以及一个附带了正确坐标的检测星表。我们利用这些来计算天体测量指标，包括尺度变化和位置差异。

![](img/6f69786f046b1c5ffed49a1153d767d1_60.png)

## 从 ZTF 项目中获得的经验教训

上一节我们介绍了如何利用 Astropy 解决天体测量问题，本节我们来总结一下从中获得的经验。

![](img/6f69786f046b1c5ffed49a1153d767d1_62.png)

Astropy 提供了许多实用工具来方便地管理 SCAMP 的输入。进行天体测量先验拟合时，最好在切平面内工作，将所有内容转换到该平面进行拟合。尽可能消除网络调用。我们在一个包含 66 台机器的 Slurm 集群上运行离线处理，可能同时运行 1000 个作业，因此网络调用是一个实际问题。我们预取了盖亚星表，并将其转换为 SCAMP 求解器需要的格式。我们还绕过了某些网络调用，例如 `astropy.coordinates.EarthLocation.of_site` 在调用时会进行网络查询以查找帕洛玛的位置，我们对此进行了硬编码。

天空偏移坐标系对于避免赤经在 360 度到 0 度跨越的问题非常有用。通常，因为我们总是在大约 1 度的天空区域工作（这是我们一个象限图像的大小），所以在局部坐标系中工作非常有用。

最后，需要注意配置文件不兼容的问题。我们曾遇到一个问题，系统中同时存在 Python 2 和 Python 3 脚本。当我们运行大量作业时，一个配置文件会随机损坏。原来是因为我们在 Python 2 和 Python 3 的虚拟环境中安装了不同版本的 Astropy，一个版本会覆盖另一个，最终导致文件损坏。解决方案是升级我们的 Astropy 版本。这也提醒我们要避免技术债务，应该下定决心将所有内容升级到 Python 3。

## 起源空间望远镜光谱模拟工具

![](img/6f69786f046b1c5ffed49a1153d767d1_64.png)

上一节我们总结了 ZTF 项目的经验，本节我们来看看 Astropy 在另一个项目——起源空间望远镜模拟工具中的应用。

![](img/6f69786f046b1c5ffed49a1153d767d1_66.png)

起源是一颗计划于未来发射的红外空间天文台。它正与 NASA 的另外三个概念一起进行研究。我们希望为该任务上的中分辨率光谱仪构建一个模拟工具，以研究望远镜孔径、到恒星/星系的距离、积分时间和光谱分辨率之间的权衡关系。我们选择 Plotly Dash 来实现这个工具。

以下是 Astropy 单位和 Astropy 常量在跟踪单位转换方面发挥了巨大作用的地方。我们从仪器首席研究员那里获得了一个计算仪器灵敏度的理想化例程，其中包含许多“魔法数字”和关于单位（如 GHz、erg/GHz 等）的注释。使用单位包极大地帮助了我们避免手动跟踪这些单位。我们还能在代码中加入断言，以确保在得到无量纲的信噪比时，我们确实得到了所需的结果。因此，我强烈建议使用单位包，它对我们帮助很大。

最近我了解到，YT 社区推出了一个独立的单位包，因为 Astropy 社区曾讨论过单位功能具有通用性，可以拆分到自己的包中。YT 社区实际上已经做到了这一点，Nathan Goldbaum 等人约四周前在 arXiv 上发表了一篇相关论文。我认为 Astropy 项目有兴趣与 YT 人员讨论如何在此基础上进一步发展，因为我们希望让事情的处理方式清晰明了。

![](img/6f69786f046b1c5ffed49a1153d767d1_68.png)

对于河外光谱的红移，Astropy 宇宙学模块提供了很大帮助，它包含了普朗克 2015 宇宙学模型，我们用它来获取光度距离随红移变化的函数。对于光谱插值，这比我想象的要困难。我原以为会有其他现成的工具可用，但结果发现 PySynphot（虽然不是 Astropy 包，但其主要维护者是 Astropy 开发者）提供了重新插值光谱的实用工具，这对本项目至关重要。

![](img/6f69786f046b1c5ffed49a1153d767d1_70.png)

我有一段该工具的在线演示视频。在星暴星系模式下，我们可以选择红移，将其推到红移为 5。我们可以将积分时间从 10 分钟改为 1 小时。我们可以改变背景。我们可以降低光度，然后可以放大查看。对于每条谱线，我们都有提示信息，给出谱线的静止波长、元素种类、信噪比和灵敏度。我必须说，我对 Plotly Dash 非常满意，它效果很好。

上一节我们介绍了模拟工具，本节我们来总结一下从中获得的经验。
