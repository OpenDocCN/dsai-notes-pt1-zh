## 智能体循环 🔄

上一节我们介绍了课程概述，本节中我们来看看智能体循环的核心概念。

智能体循环是一个循环迭代的环境，大型语言模型（LLM）在其中执行特定时长。它始于我们一直在讨论的“组装上下文”步骤。然后，我们将组装好的上下文传递给LLM，使其能够对接收到的信息进行推理和决策。接着，LLM会采取一个行动。这个行动可以是响应用户、调用工具，或是向用户请求更多输入。

总而言之，这就是我们所说的智能体循环：我们组装上下文、调用LLM、并根据LLM的响应采取行动。它是循环且迭代的，意味着这个循环可以执行特定时长。循环的终止通常由一个停止条件决定，例如LLM给出最终答案、目标完成，或者循环执行中出现错误或超时。

智能体循环在用户向循环提供上下文或输入时开始，并启动上下文组装过程。

![](img/0bf67976c04b11b73a7443e9ad4d086e_3.png)

更明确地说，智能体循环是一个执行特定次数的循环环境，其中LLM被调用，其响应被观察，并可以采取行动。它组装、调用、行动，直到满足停止条件。

![](img/0bf67976c04b11b73a7443e9ad4d086e_5.png)

在代码中，智能体循环如下所示：

![](img/0bf67976c04b11b73a7443e9ad4d086e_7.png)

```python
max_steps = 10
for step in range(max_steps):
    # 组装上下文
    context = assemble_context()
    # 调用LLM
    response = llm.invoke(context)
    # 根据响应采取行动
    action = take_action(response)
    # 检查停止条件
    if is_stop_condition_met(response):
        break
final_result = extract_result(response)
```

## 循环内外的记忆操作 💾

现在我们已经理解了什么是智能体循环，接下来可以探讨循环内外的记忆操作。

让我们从循环外的记忆操作开始。在之前的课程中，我们概述了不同形式的智能体记忆，以及存储在数据库表中的相应记忆存储。这些表通过一个内存管理器暴露了我们可以调用的读写操作。这些操作在智能体循环中显示和使用，但具体是在循环之外执行的。

我们引入了一个“智能体框架”的概念，因为循环内外的记忆操作和其他操作共同构成了智能体框架。这些程序化的脚手架使得AI智能体能够可靠地执行。

在启动条件（即将进入智能体循环时），我们通常会有从记忆中读取的操作，因为我们需要为LLM组装上下文。

但是，我们也可以有在智能体循环内部执行的内存操作。这是因为我们的智能体可以调用工具，同时也有一些工具是在智能体控制之外、通过编程方式执行的。

![](img/0bf67976c04b11b73a7443e9ad4d086e_9.png)

以下是这些操作的示例：
*   **循环外的操作**：在启动时从长期记忆（如对话记忆、知识库）中读取以组装初始上下文。
*   **循环内的操作**：
    *   **智能体触发的工具**：例如搜索互联网、扩展摘要。
    *   **确定性编程操作**：例如，当上下文窗口使用量超过其容量的80%时，自动触发对话摘要。

这些循环内外的操作共同构成了你的框架的一部分，从而打造出一个可靠的、具备记忆意识的AI智能体。

## 实现智能体循环 🤖

![](img/0bf67976c04b11b73a7443e9ad4d086e_28.png)

现在我们可以真正进入智能体循环了。

![](img/0bf67976c04b11b73a7443e9ad4d086e_30.png)

在主智能体循环中，我们将首先定义一个名为`call_agent`的函数，它将接收用户查询（这将是实际触发智能体循环的启动条件）。我们还将指定一个线程ID（因为我们希望这会话具有作用域），以及我们希望智能体循环迭代的最大次数。

![](img/0bf67976c04b11b73a7443e9ad4d086e_32.png)

在这一部分，我们识别了在智能体循环外和循环内调用的记忆操作。我们将从循环外的记忆操作开始。

以下是循环外用于构建初始上下文的记忆操作步骤：
1.  构建对话记忆。
2.  构建知识库记忆。
3.  构建工作流记忆。
4.  构建实体和摘要记忆。

下一步是执行一个确定性的记忆操作：检查上下文使用量是否超过或等于80%。如果上下文窗口的使用量大于80%，我们将以编程方式触发摘要。

最后，我们将整个上下文赋值给一个变量`context`。

接下来的步骤是检索与查询相关的工具。同样，这个读取操作是智能体循环外的记忆操作。这减少了我们需要提供给智能体的工具数量，因为我们只需要提供与传入智能体循环的特定查询相关的工具。

然后，我们将写入对话记忆：查询、角色和线程ID。

![](img/0bf67976c04b11b73a7443e9ad4d086e_34.png)

现在，在进入智能体循环之前，我们做的最后一步是创建将要传递给智能体循环内LLM的消息或上下文。这个上下文将包含使LLM更具记忆意识的智能体系统提示词，以及我们开始时已有的角色和上下文。

在智能体循环中，我们使用之前探讨过的`call_openai_chat`函数。我们将传入消息和可供智能体调用的工具。将LLM生成的响应赋值给一个名为`response`的变量。

通过分析响应中的消息，我们可以决定是进行工具调用操作，还是直接转向最终响应。记住，在智能体循环中，智能体可以决定采取行动或响应用户（满足停止条件）。

如果智能体决定进行工具调用，我们会将其附加到消息中。在智能体循环内，我们能够执行工具，并传入智能体指定执行的参数和名称。

![](img/0bf67976c04b11b73a7443e9ad4d086e_36.png)

现在，在智能体循环内，我们将进行的一项记忆操作是将一些工具日志移出循环并存入数据库。这就是上下文卸载。

![](img/0bf67976c04b11b73a7443e9ad4d086e_38.png)

一旦收到工具结果，智能体循环的迭代次数将增加，然后我们将向LLM提供工具结果。如果LLM决定中断并给出最终答案，我们将显示它并将其赋值给`final_answer`变量。

还有一种情况是跳出智能体循环，即达到最大迭代次数但仍未得到最终答案。然后，我们将给`final_answer`一个预设的模板响应，说明智能体无法完成请求。

![](img/0bf67976c04b11b73a7443e9ad4d086e_40.png)

在智能体循环外，我们仍然有一些记忆操作，例如写入工作流记忆、写入实体记忆和写入对话记忆。这些是为了捕获智能体执行过程中的一些成果，例如执行查询或实现目标所采取的步骤数（我们可以将其存储为工作流记忆），以及此特定步骤的结果。我们还会将最终答案中检测到的任何实体写入记忆。

最后，我们将最终答案和助手角色写入对话记忆。

这就是智能体循环及其内外操作的概述。具体来说，我们有了一个可以调用的记忆感知智能体，它能够利用记忆和连续性执行操作。

对于对话记忆，我们使用Markdown子标题标识符来标识它。我们使用Markdown标题使LLM能够理解上下文窗口内信息的层次结构。LLM能够理解这一点，因为此类或类似结构的信息是训练数据的一部分，因此它们具有理解Markdown文件中结构和层次信息的潜在能力。

在对话记忆中，我们指定了智能体应如何使用此记忆以及何时利用它。这使我们的LLM具备了记忆意识。我们对上下文窗口中包含的每种记忆（从知识库到实体记忆等）都这样做。
