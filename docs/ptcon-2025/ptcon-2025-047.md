在本节课中，我们将学习Monarch，这是一个由PyTorch团队创建的、用于构建分布式机器学习系统的Python原生编程框架。我们将了解其核心设计理念、关键特性，并通过实际应用案例展示其如何简化大规模分布式系统的开发。

## 概述：大规模机器学习的挑战与Monarch的解决方案

大家好，我是Zach，这是Colin，我们将为大家演示Monarch。

过去几年，大规模训练模型带来了独特的系统挑战。例如，预训练需要协调成千上万个GPU，高效利用GPU的策略会随着模型规模或数据中心规模的变化而改变。一旦规模足够大，单个组件的故障就可能导致整个作业死锁。

后训练阶段并未变得更容易，因为它结合了预训练的复杂性与多个组件的复杂异步性。你的预训练器变成了后训练模型中的学习器，但你现在还需要生成器来产生新的训练轨迹、执行任意任务（如浏览网页、编码或模拟游戏）的环境，以及用于快速传输权重以保持策略一致性的参数服务器。

在这个领域的竞争优势来自于能够快速编程这些设置并尝试新想法。为了帮助解决这类问题，我们创建了名为Monarch的框架。从根本上说，Monarch是一个用于构建分布式机器学习系统的Python API。它是一个命令式API，其基本设计原则是：一切都应该是Python API。这是PyTorch的创立基础，我们希望将其带入为机器学习创建分布式系统的世界。

## 核心概念：一切皆Python API

当我说“一切都是Python API”时，我指的是：它是一个用于获取机器、在这些机器上启动进程、然后让它们使用Actor或远程直接内存访问（RDMA）快速相互通信的Python API。Monarch的理念是，单个程序定义了整个ML系统。这样做有助于最大限度地提高研究或生产效率，因为它让尝试新想法变得非常容易。这基本上就像将集群编程当作单台机器一样。例如，你可以在笔记本中运行Monarch，就像本次演示中展示的那样。

## 初探Monarch：一个“Hello World”示例

那么，实际使用Monarch是什么样子呢？以下是一个类似“Hello World”的Monarch代码片段。

首先，我们在Monarch中获取机器。这里我们只使用我的本地开发服务器，并将其称为“trainers”。然后，我们可以使用Python API在这些机器上启动进程。这里我们启动了8个GPU进程，并在每个进程上放置一个执行训练的“learner” Actor。目前，我们只是传入一个假的数据加载器。启动后，我们得到了8个进程，它们都报告了一些日志信息。

```python
# 示例：获取主机并启动Actor进程
hosts = monarch.acquire_hosts(‘local’)  # 获取本地主机
trainers = hosts.named(‘trainers’)  # 将主机命名为“trainers”

# 在trainers上启动8个GPU进程，并放置Learner Actor
learners = trainers.launch(8, gpu=True, actor=Learner, data_loader=fake_loader)
```

这为你提供了一个命令式API来将Actor启动到主机上。

消息传递对于任何其他类型的Actor系统都很熟悉。你可以调用Actor上的方法，这里我调用`step`方法。Monarch的一个独特之处在于，你从不只与单个Actor通信。我们启动了8个进程，因此实际上有8个Actor，它们都会响应。稍后我们会详细讨论这一点。

如果我们看看这个Actor类是什么样子，对于任何在其他框架中使用过Actor的人来说都会很熟悉。你有一个继承自Actor基类的类，可以为该类编写初始化方法，并定义我们称为“端点”的东西，也就是你可以向Actor发送消息的方法。`step`就是其中一个端点。

## 核心抽象：Mesh（网格）

我之前已经提到过Monarch的一个独特之处：我们不将任何东西作为单例处理。你不是发送单个消息，相反，Monarch中的一切都是实体的**组**。你有Actor组、进程组或主机组，我们称这些组为**Mesh**。它们基本上是具有命名维度的多维数组，有点像张量，但你的张量包含的是主机、进程或Actor。

因为一切都是用这些Mesh实现的，所以可以非常容易地在单主机和多主机之间转换Monarch程序，因为你可以在定义作业时直接请求更多主机。之前我使用API将进程附加到本地机器，但如果你在集群上工作，可能有一个像SLURM这样的调度器。我们也有API可以让你通过SLURM获取主机。这样，你就可以请求2000个GPU，获得一个长度为2000的主机Mesh，然后将其输入到之前使用的完全相同的程序中，它将在那些主机上工作，并且仍然可以在笔记本中运行。唯一改变的是你获取主机的方式。我认为这很好地展示了当一切都是Python API时你能做什么。

## 可扩展的消息传递

但是，假设我们切换到2000台主机。如果你只是天真地实现这一点，你会遇到麻烦，因为从一台机器告诉2000台主机执行一步操作非常困难，耗时太长，不切实际。因此，作为Mesh实现的一部分，我们还为Mesh提供了优化的消息传递。Mesh通过使用类似日志和广播树的结构来传递所有消息，从而实现可扩展的消息传递。顶层客户端只发送恒定数量的消息，这些消息会像树一样展开。

除了这种消息广播，我们还有结果聚合，也可以进行日志记录。很多时候，当你试图协调时，你会说“执行模型的一步”或“现在是时候将权重从学习器更新到生成器了”，并且我需要你确认操作正确完成。但返回值总是相同的，因此我们在返回时对公共值使用压缩进行聚合。这样，我们也可以使对客户端的响应是日志式的。这基本上让Monarch的消息传递可以扩展到数千个Actor。你不需要使用笨拙的屏障原语，你可以直接用消息传递和等待响应来定义一切。

## 可组合性与进程图构建

我们之前展示的所有这些对象，如主机、进程和Actor，都是Python API的一部分，它们是一等对象，你可以在消息中发送它们。这意味着你可以在系统中移动它们。这让你能够以可组合的方式构建可以在系统中移动的Actor，而无需离开笔记本或单个进程。

例如，在我们之前展示的例子中，我们有一个假的数据加载器。但如果你想要一个真正的数据加载器，你可能会定义一个执行数据加载的Actor，并为该数据加载器生成单独的进程。因为数据加载器是一等对象，你可以简单地将对数据加载器的调用放入我们最初传递给训练器以加载数据的回调中，然后将其传递给训练器。你会注意到，在这种情况下，我们根本没有改变训练器。训练器甚至不需要知道有一个Actor参与加载数据。这让我们能够通过仅对代码进行本地编辑来创建更复杂的进程图。

## 容错机制：分层监督

一旦你有了这个系统图，我们在Monarch中内置了许多在构建这些RL系统时非常需要的东西，这样你就不必过多考虑大规模时的常见问题。大规模的一个主要问题是容错性。你的作业会以各种方式中断，特别是如果你在做任意环境模拟，这些环境经常会出错，而你不希望整个作业停止。因此，我们创建了这个Actor系统，这些Actor是分层创建的。你看到有一个顶层客户端启动更多Actor，它们可能启动更低层的Actor和进程。这基本上产生了一个启动其他Actor的Actor层次结构，我们也通过这个层次结构处理容错。

其理念是我们拥有所谓的“渐进式容错”。如果你什么都不做，那么如果学习器出现错误，它将沿着这棵树向上冒泡，就像代码中的异常冒泡一样。如果没有处理程序，它将到达根Actor，你的作业将停止。这已经比现在常见的情况要好，比如使用`torch.multiprocessing`时，很容易让某个角落的生成器失败，然后整个系统死锁，而不是报告出了问题。

但这种“重启一切”的方式效率不高。例如，像评分器这样只是运行模型对轨迹进行评分的东西，并不是真正有状态的，如果它中断了，你实际上可以继续训练。在这些情况下，我们允许你自定义如何处理错误消息。你可以在父Actor上定义一个名为`supervise`的特殊魔术方法，它本质上会捕获异常。这让你知道发生了异常，然后你可以提供自己的策略来处理该异常。在这种特定情况下，你可以决定放弃那个评分器，但在其位置启动一个新的，并让它继续运行。这将停止错误传播，你的其他训练器可以继续。

## 核心服务：基于RDMA的点对点数据传输

到目前为止，我们讨论了Monarch的核心系统，我们通过Actor和容错提供可扩展的消息传递。事实证明，这两个关键组件使得构建许多强大的服务变得容易得多。因此，我想谈谈基于这个消息传递系统，Monarch现在附带的两项强大服务。

第一项服务是我们称之为通过RDMA（远程直接内存访问）进行的点对点数据传输。

Monarch中的这项服务可以将任何CPU或CUDA内存转换为零拷贝的RDMA缓冲区句柄。任何拥有这些句柄之一的Actor都可以远程读取或写入该内存。这在系统图中意味着什么？以学习器在运行开始时将权重传递给推理引擎为例。你可以用学习器拥有的所有权重创建RDMA缓冲区，然后将这些RDMA缓冲区的引用（即读写该内存的句柄）发送给推理引擎。随这些句柄一起，你可能还会制定一个计划，即每个推理引擎应该从哪里读取，以便在学习器之间良好分布，以及如何更新权重。然后，在训练后期，你只需广播一条消息，说“是时候同步你的权重了”。此时，推理引擎可以使用RDMA读取直接从学习器中读取权重，而不需要再次协调地址，它可以立即访问网卡并开始操作。

与类似NCCL的方案相比，这种方式有很多优势。首先，这些事务是单边的。如果你有一个引用，你不需要对方发出发送指令，只要你有引用，就可以直接读取或写入数据。因为你不需要通过NCCL协调这些连接的启动，我们可以使设置非常快。新连接的设置大约只需几毫秒，并且不需要预先在事物之间分配通信器。内存注册同样非常快，在进程的生命周期内基本上可以摊销到免费。这样，你仍然可以获得与NCCL等先前技术相当的速度。我还要说，这具有更好的容错特性，因为如果一个设备坏了，你不会破坏整个进程组，你只是无法再向那个损坏的设备发送消息，那些事务将无法完成，但你仍然拥有其他所有设备的工作事务。

因此，我们希望这是开始能够为核心服务提供基础，以构建更高效的RL系统。我们已经在Torch Forge中看到了一些例子，例如基于此RDMA技术构建的分布式对象存储。

## 核心服务：单控制器分布式张量

可以从Monarch消息传递系统中衍生出的第二类功能或服务是**单控制器分布式张量**。

Monarch非常擅长向许多地方广播消息，这使我们能够指示许多不同的进程执行张量计算。如果你在Monarch中生成进程，就像我们这里的两个GPU进程，然后你调用一个名为`Activate`的上下文管理器，那么在该上下文管理器下发生的任何PyTorch计算都将在Mesh中的每个设备、每个进程上发生。因此，这里的`RA`实际上是一个跨整个计算实例的分布式张量。然后，你可以接着使用内置的集合操作（不是集合通信，因为只有一个控制器）进行归约，例如跨Mesh的GPU维度进行all-reduce。目前，这对于调试和尝试想法非常有用。未来，我们希望提高其性能，使其足以仅用在此张量类型上定义的并行算法来编写整个分布式训练栈。

## 总结：Monarch的核心价值

回顾一下，我介绍了Monarch的许多不同特性，但我们认为真正定义我们在这里提供的能力的是：**仅通过Python API，从单个进程构建分布式ML应用程序**。独特的是，作为其组成部分，我们提供了诸如通过Mesh实现的可扩展消息传递（这样你就不必担心需要与多少个Actor通信）、通过监督树实现的内置容错，以及独特的服务，如点对点RDMA传输和单控制器分布式张量。

## RDMA性能与应用

首先，Zach详细介绍了Monarch的RDMA。它支持Actor之间（无论是在CPU还是CUDA内存上）的直接内存访问。正如Zach提到的，这是零拷贝、单边且完全异步的，这意味着Actor Mesh中任何拥有RDMA缓冲区句柄的Actor都可以在没有CPU同步的情况下直接读取或写入另一个Actor的内存。Zach已经介绍了它带来的一些好处。我们看到，无论是同一主机内的GPU到GPU、跨GPU还是跨机架内主机，其性能都达到了网卡性能的99.5%。这使得它非常适合权重流式传输、参数分片或服务互连等应用。我们很高兴能让用户不将其视为分布式系统问题，而只是通过这种数据流将其视为Monarch中的另一个Actor调用。

## 与现有框架集成：vLLM与TorchForge

我们想做的第一件事就是与当今非常流行的异步RL框架vLLM集成（我相信他们也在会议上，你们很多人可能熟悉）。我们想验证Monarch是否可以轻松协调这个异步RL框架中的控制流。我们用它来训练Quin 2.57B（一个使用GRPO的数学推理模型）。我们将其从16个H200扩展到2048个H200。这些运行稳定，并实现了可比的吞吐量和收敛性。此外，正如Zach提到的，由于Monarch具有这种监督层次结构和可扩展性优势（我们看到这种日志广播以及结果聚合），这是使用Monarch进行通信的一大好处。

此外，正如我们昨天宣布的，我们对TorchForge感到非常高兴，该团队昨天做了一个精彩的演示。TorchForge是原生基于Monarch从头构建的PyTorch开源异步RL库。它包含了几个重要的Monarch原语。

我认为这里的关键思想是：**没有协调代码，没有重试逻辑，没有显式的消息传递，只有纯粹的异步Python**。在底层，Forge使用Monarch的两个主要部分来协调控制流和数据流。在控制流方面，Forge将每个RL组件都视为一个服务，这些服务在Monarch的Actor Mesh上实现。它们实现了负载均衡、容错的组件以及它们之间的协调。PyTorch团队还构建了TorchDo，这是一个主要用于存储张量的分布式键值存储，它由Monarch的RDMA和RDMA缓冲区支持。

如果你还没看过，我们鼓励你去GitHub和博客文章上查看它们，他们也很乐意收到你的反馈。

## 与TorchFT集成实现弹性训练

Monarch的另一个集成是与TorchFT。正如你们许多人可能熟悉的，在大规模场景下，硬件和网络故障并不罕见，而是非常常见。TorchFT帮助解决这个问题，使训练具有弹性。它通过将训练作业拆分为独立的副本来实现。在副本之间，梯度通过all-reduce进行同步。但重要的是，在发生故障时，每个副本都作为自己独立的故障域，这使得健康的副本即使在某个副本失败时也能继续取得进展。

Monarch与此很好地配合，因为我们可以非常快速地检测故障，隔离失败的副本组，并仅重启那些进程。这一切都得益于Monarch的分层监督层次结构。

在实验中，我们进行了一个30节点、240 GPU的训练作业，通过TorchTitan训练Quin 3 320亿参数模型。我们注入了100次故障，包括段错误、进程终止、NCCL中止甚至主机驱逐。实验表明，我们对这些进程故障实现了92%的恢复率，对机器故障实现了2.5分钟的恢复时间，这比单纯通过SLURM重启快了约60%。这种策略在GPU数量达到10万甚至百万级别时至关重要。
