在本节课中，我们将学习 Anthropic 博客文章《构建高效智能体》中提出的核心概念。我们将从零开始构建文章中提到的每一种工作流和智能体模式，并解释它们是什么、如何构建以及各自的实用价值。

## 工作流与智能体的定义 🧩

理解工作流与智能体的区别至关重要。我们可以用一个简单的方式来思考它们的定义。

工作流可以看作是在大语言模型调用周围搭建的、具有预定义代码路径的“脚手架”。这种方式已经流行多年，在许多情况下，将大语言模型调用嵌入到一组固定的代码路径中是非常合理的。

有时，你甚至可以让大语言模型自己决定在工作流中走哪条路径，这属于我在这里画的中间类别。我们稍后会详细讨论所有这些模式。但这大致是直观的理解：工作流就像是带有预定义代码路径的脚手架，你可以将大语言模型调用嵌入其中。

智能体则移除了脚手架。你基本上是让一个大语言模型自主指导其行动。在这种情况下，行动通常是工具调用。大语言模型会接收来自这些工具调用的反馈，并决定下一步做什么。

需要指出的是，你当然可以在工作流中让大语言模型执行工具调用。但区别在于，工作流周围有脚手架，而智能体是无界的。智能体直接接收来自工具调用的环境反馈输出，并决定下一步行动，周围没有那种推理脚手架。这就是真正的区别。

![](img/6160fd08f325f189c794e8d28c01b46c_3.png)

## 为什么需要框架？⚙️

在开始之前，有必要说明一下为什么需要框架。实现这些模式并不一定需要框架，有时用几行代码就能完成。我完全理解这一点。所以你可能会问，为什么要讨论这个？为什么要推广框架？

我的看法是这样的。LangGraph 的目标是**最小化实现这些模式的开销**。它不抽象提示词，也不抽象架构。LangGraph 真正提供的是**支撑任何工作流或智能体的底层基础设施**。这是关键点。

具体来说，它主要提供三样东西：

1.  **持久化**：这为你提供了记忆功能，以及人机交互循环的能力。例如，你可以在智能体处理过程中暂停，并批准一个工具调用，或者中断智能体的操作。这在许多情况下极其有用。
2.  **流式传输**：当然，大语言模型 API 本身可以流式传输令牌。但当你构建这些工作流或智能体时，有时你希望流式传输特定步骤的输出，例如工具调用的输出。你通常希望对流式传输有比单纯的大语言模型调用输出更多的控制权。LangGraph 让你能很好地控制从工作流或智能体输出的内容，这在构建并将其投入生产时非常有用。
3.  **部署**：测试、调试和部署。这是使用 LangGraph 的一大优势。从任何工作流或智能体的实现到部署，都变得极其简单。

LangGraph 不抽象提示词，也不抽象架构。它真正提供的是位于任何这些工作流或智能体之下的底层基础设施。

## 工作流模式：提示链 🔗

![](img/6160fd08f325f189c794e8d28c01b46c_24.png)

接下来是博客文章中涵盖的一些工作流模式，从提示链开始。

以下是直观理解：每个大语言模型调用处理前一个调用的输出。当你想要这样做时，你可以将任务分解为几个不同的大语言模型调用。

在这个小例子中，他们展示了一个输入，调用1，你可以对调用1的输出进行一些门控判断，输出进入调用2，那个输出再进入调用3。

![](img/6160fd08f325f189c794e8d28c01b46c_26.png)

![](img/6160fd08f325f189c794e8d28c01b46c_27.png)

![](img/6160fd08f325f189c794e8d28c01b46c_28.png)

让我们从零开始构建这个，并做一个例子：我们构建一个链，它接收用户的一个主题，大语言模型讲一个笑话，我们检查这个笑话是否有笑点，然后用两个后续的大语言模型调用改进它两次。

![](img/6160fd08f325f189c794e8d28c01b46c_29.png)

![](img/6160fd08f325f189c794e8d28c01b46c_30.png)

![](img/6160fd08f325f189c794e8d28c01b46c_31.png)

![](img/6160fd08f325f189c794e8d28c01b46c_33.png)

当我在 LangGraph 中工作时，我基本上需要做的就是为我想要在工作流中修改的所有内容定义一个容器。在这个特定情况下，我将创建一个字典，它将包含我们从用户那里获取的 `topic`，第一个调用的输出 `joke`，第二个调用的输出 `improved_joke`，以及最终调用的输出 `final_joke`。

在布局工作流时，我喜欢把它们画出来。博客文章已经把这些都画出来了，这很酷。我们画出来，想想看，例如，每个调用或步骤只是一个不同的函数。

所以对于这个特定的工作流，我将创建三个函数：`generate_joke`， `improve_joke`， `polish_joke`。这些都将只是简单的大语言模型调用。

有趣的是，在 LangGraph 中，我定义的这个容器或状态会传递到每一个步骤中，并且我可以从中提取任何我想要的东西。在这个例子中，我通过 `state[“topic”]` 来获取用户写入状态的主题。我可以通过返回这个 `joke` 键（即我的大语言模型调用的输出）来将内容写回状态。对于 `improve_joke`，我填充 `improved_joke`。对于 `final_joke`，我填充 `final_joke`。

![](img/6160fd08f325f189c794e8d28c01b46c_35.png)

所以发生的事情是，这些步骤在进行大语言模型调用，并且我正在用每个大语言模型调用的输出填充我的工作流状态或容器。这就是全部过程。

现在，注意在这个工作流中，这里有一个门控。所以在这个例子中，我将创建一些门控逻辑。有趣的是，我可以接收状态，然后检查笑话是否包含问号或感叹号。这只是一些任意的标准。然后我可以传递任何我想要的东西到这里，只是两个字符串：“pass” 或 “fail”。这可以用作 LangGraph 中的条件边或门控。

![](img/6160fd08f325f189c794e8d28c01b46c_37.png)

![](img/6160fd08f325f189c794e8d28c01b46c_38.png)

现在，我有一个包含我想要在工作流中修改的所有内容的容器。我已经将工作流中的每个步骤定义为一个独立的函数。我定义了一个门控，它将作为对笑话输出的检查。

![](img/6160fd08f325f189c794e8d28c01b46c_39.png)

![](img/6160fd08f325f189c794e8d28c01b46c_40.png)

现在我在 LangGraph 中将其布局为一个简单的工作流。我需要做的就是接收我的状态，初始化工作流，将这三个步骤添加到其中。我们添加了 `generate_joke`， `improve_joke`， `polish_joke`。这是我定义图或工作流连接性的地方。

我将从 `generate_joke` 开始。然后我想检查第一个调用的输出，看它是否有笑点。所以我添加一个条件边来连接 `generate_joke`。根据这个函数的输出，如果是 “pass”，我就去 `improve_joke`（匹配节点）。如果是 “fail”，我就直接结束。然后我创建从 `improve_joke` 到 `polish_joke`，再从 `polish_joke` 到结束的边。将其编译为一个工作流。就这样。

这就是他们在博客文章中画出的完全相同的、应用于笑话生成的工作流。一旦我有了这个工作流，我就可以非常简单地运行 `chain.invoke()`。这是在 LangGraph 中调用任何链、工作流或任何东西的非常简单的方法。我传入用户的一个主题。我们可以看到大语言模型创建了一个初始笑话，改进了它，并创建了最终笑话。这是一个非常简单的链的例子。
