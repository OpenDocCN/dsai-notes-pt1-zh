## 概述
在本节课程中，我们将学习面向 GPTU 架构的 FlagTree 编译优化实践。我们将从编译器的整体架构入手，了解如何添加一个优化 Pass，并探讨从多个角度进行编译优化的思路。

## 编译器整体架构介绍
上一节我们介绍了本次分享的主题，本节中我们来看看 FlagTree 编译器的整体架构。一个程序要写得简洁且运行高效，除了编程语言本身，编译器是核心部分。

FlagTree 主要由两部分构成：
1.  **编程语言**：采用 Python 风格的编程，让用户能更无感知地编写并行计算逻辑。它可以指定线程块的划分方式及不同的数据读写模式。
2.  **编译器**：负责将高级语言代码转换为高效的机器码。

以下是其编译流程架构：
*   **前端**：接收用户编写的 Triton Kernel 代码。
*   **AST 解析**：将代码解析为 Python AST，再转换为统一的硬件无关中间表示 **`TritonIR`**。
*   **中间优化**：在 `TritonIR` 层进行内联、规范化等优化。
*   **硬件相关降级**：将 `TritonIR` 降级到硬件相关的 **`TritonGPUIR`**。在这一层，会进行访存合并、数据张量布局、共享内存优化等核心优化。
*   **后端代码生成**：将 `TritonGPUIR` 进一步降级为 **`LLVMIR`**，最终由基础编译器（如 LLVM）生成可执行的二进制代码。

整个流程表明，FlagTree 编译器采用统一的中间表示和优化机制，支持多种国产芯片，打通了从 AI 编程到传统编译的链路。

## 理解优化 Pass 与 Pass 管理器
我们刚才聊到了优化流程，现在来深入理解其中的核心组件：优化 Pass 和 Pass 管理器。

**优化 Pass** 是指对程序中间表示进行分析或转换的一个独立阶段。主要分为两类：
1.  **分析类 Pass**：只收集信息，不修改代码。
2.  **转换/优化类 Pass**：通过修改和重写 IR 来进行代码优化。

将多个优化 Pass 组织起来的形式，就叫做 **Pass 管理器**。它的作用是确定 Pass 的执行顺序、维护 Pass 间的依赖关系，并确保一个 Pass 的输出能作为下一个 Pass 的输入。

不同编译器的 Pass 管理器有差异：
*   **LLVM**：ass 顺序固定，且只负责中间层优化，不管理降级过程。
*   **TVM**：ass 组合不固定，存在寻找最优解组合的问题。
*   **Triton 编译器**：模块化清晰。以下是其 Pass 管理器代码结构示例：
    ```cpp
    // 伪代码结构示意
    void runPassManager(Module& module) {
        runPass(TritonIRToTritonGPUIRPass); // TritonIR 到 TritonGPUIR 的转换
        runPass(TritonGPUIROptimizationPass); // 优化 TritonGPUIR
        runPass(TritonGPUIRToLLVMIRPass); // TritonGPUIR 到 LLVMIR 的转换
    }
    ```
Pass 管理器在 Triton 编译器的 `compile.py` 文件中。

## 如何添加一个优化 Pass
在了解了优化 Pass 的基础概念后，我们来看看如何为编译器添加一个新的优化 Pass。

以下是添加一个优化 Pass 的标准工作流程：

**1. 在 `Pass.td` 文件中定义元数据**
首先，需要在 TableGen 文件（如 `Pass.td`）中定义 Pass 的相关元数据，例如名称、描述和构造方式。不同版本格式略有差异，新版本可通过 TableGen 直接生成头文件。
```tablegen
def MyNewPass : Pass<"my-new-pass"> {
  let summary = “这是一个新的优化Pass示例”;
  let constructor = “createMyNewPass()”;
}
```

**2. 实现优化 Pass 的核心逻辑**
这是优化的核心步骤，即创建 C++ 类来实现具体的优化逻辑。例如，实现一个将加法操作替换为更高效指令的 Pass。
```cpp
class MyNewPass : public PassInfoMixin<MyNewPass> {
public:
  PreservedAnalyses run(Operation *op, AnalysisManager &am) {
    // 遍历模块中的指令
    op->walk([&](Operation *innerOp) {
      // 定位特定的操作类型，例如 “arith.addf”
      if (isa<arith::AddFOp>(innerOp)) {
        // 在此处实现优化逻辑，例如替换操作
        // ...
        // 调试时可使用 llvm::errs() 打印信息
        llvm::errs() << “Found an add operation: ” << *innerOp << “\n”;
      }
    });
    return PreservedAnalyses::all();
  }
};
```

**3. 在 `CMakeLists.txt` 中添加源文件**
完成 Pass 实现后，必须在编译系统的 `CMakeLists.txt` 文件中加入对应的源文件，否则编译器会找不到该文件。

**4. 修改 `Pass` 注册代码**
在 `Pass.cpp` 或 `PassRegistration.cpp` 文件中，将 C++ 实现的 Pass 暴露给编译器框架的接口。这一步确保 Pass 能被编译器识别和调用。

**5. 修改 Pass 管理器**
最后，将新 Pass 添加到合适的优化流水线中。例如，如果这是一个 `TritonGPUIR` 层的优化，就需要将其插入到 `TritonGPUIR` 的优化阶段序列中。

完成以上步骤后，一个新的优化 Pass 就添加成功了。在调试时，可以使用 `-mlir-print-ir-after-all` 或 `-debug-only=my-pass` 等编译参数来输出 Pass 优化前后的 IR 变化，便于验证效果。

## 编译优化实践：四个思考角度
我们已经掌握了添加优化 Pass 的方法，接下来可以思考从哪些角度开展编译优化工作。本节我们将探讨四个主要方向。

### 角度一：针对编译器原生优化的不足
我们可以从编译器现有优化存在的缺陷或不足入手。例如，**访存合并优化** 是提升全局内存访问效率的关键技术，它涉及两个重要概念：
*   **缓存行**：GPU 一次从全局内存加载的最小数据量，通常是 64 或 128 字节。公式表示为：`内存事务大小 = 数据总量 / 缓存行大小`。
*   **内存事务**：一次读取一个或多个缓存行的过程。内存事务越少，访存效率越高。

Triton 编译器中的 `MemCoalescingPass` 实现了该优化，主要包含轴分析、计算目标数据布局、重写 IR 三个步骤。其中，**轴分析** 计算张量的连续性、整除性、常量性等属性，至关重要。

然而，我们发现其轴连续性计算在某些情况下（如经过四则运算后）可能不正确。针对此，我们可以：
1.  添加额外的轴属性信息，修正连续性推导公式。
2.  优化数据布局参数 `size_per_thread` 的计算方式，避免在降级到 `LLVMIR` 时出现性能问题。
3.  解决 `size_per_thread=1` 时影响向量化优化的问题，实现多维度张量的连续向量化访存。

### 角度二：针对特定算子的性能瓶颈
当发现某些算子（如规约、矩阵转置）成为性能瓶颈时，可以针对它们进行优化。

*   **规约算子优化**：Triton 在处理非连续轴规约时，采用树形规约算法并使用共享内存，在某些形状下会带来较大访存开销。我们可以：
    *   添加多种规约算法实现（如块规约、warp 规约），根据问题形状选择最优路径。
    *   在新算法中引入数据打包等优化，例如将两个 `FP16` 数打包成一个 `FP32` 数进行规约，提升速度。

*   **矩阵转置算子优化**：该算子广泛用于矩阵乘法和卷积。我们可以：
    *   调整共享内存的连续存储方向。
    *   在降级过程中引入内存填充技术，降低 Bank Conflict 的频率。

### 角度三：针对特定硬件特征
针对如 **张量核心** 这类专用硬件单元进行优化。张量核心计算速度快，瓶颈常在于数据供给。

**优化思路**：
1.  **分块配置搜索**：分析张量核心操作数的形状和共享内存信息，构造多种分块配置，形成搜索空间，通过算法寻找最优解。伪代码表示其思想：
    ```python
    search_space = []
    for block_m in [64, 128, 256]:
        for block_n in [64, 128, 256]:
            for block_k in [32, 64]:
                if is_valid_config(block_m, block_n, block_k):
                    search_space.append((block_m, block_n, block_k))
    best_config = find_best_performance(search_space)
    ```
2.  **计算与访存重叠**：通过定义-使用链分析，调整计算与访存的顺序，实现流水线执行。
3.  **循环分块**：将大块计算进一步切分为子块，实现更细粒度的流水，提升数据供给速度。

### 角度四：引入通用优化方法
将传统编译器中的通用优化方法引入到 Triton 编译器中，例如 **代数变换**。它通过对程序进行数学等价变换来生成更高效的代码，常见变换包括：
*   简化表达式（如 `x * 1 => x`）
*   重排运算顺序
*   运算类型转换
*   运用结合律、交换律

**实现步骤**：
1.  构建优化 Pass，遍历 IR 及其节点。
2.  使用定义-使用链分析依赖关系，定位待变换的节点。
3.  构建新的、等价的 IR 结构。
4.  用新结构替换旧节点，并更新所有相关的定义-使用链关系，确保程序正确性不变。

## 总结
本节课中，我们一起学习了面向 GPGPU 的 FlagTree 编译优化实践。我们从编译器的整体架构讲起，详细说明了如何添加一个优化 Pass 的完整流程。接着，我们探讨了进行编译优化的四个主要角度：**针对编译器原生优化的不足进行增强**、**针对特定算子的性能瓶颈进行优化**、**利用特定硬件特征挖掘性能**、以及**引入通用优化方法**。这些思路为开展编译优化工作提供了清晰的路径。
